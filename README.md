# wazuh-lab
Домашний SOC с использованием установки Wazuh Server через Docker Compose 

Почему?Для чего? И что мне это даст?

Wazuh представляет собой полноценную XDR-платформу (Extended Detection and Response), которая из коробки предоставляет:

 - Unified Security Platform: SIEM + EDR + Vulnerability Management в одном решении
 - Нативная интеграция с MITRE ATT&CK: каждое правило привязано к конкретной тактике
 - AI Assistant для анализа: встроенная поддержка LLM для расшифровки сложных событий
 - Готовая интеграция с облаками: мониторинг AWS CloudTrail, Azure Activity Logs, GCP Audit Logs

 ИИ в SOC для автоматизации SOAR платформ -  это очень важная часть изучения в рамках моего проекта
 AI-решения на базе SOAR-платформ уже выполняют первичное обогащение алертов, автоматическую корреляцию и даже предлагают плейбуки реагирования. Звучит красиво, но на деле - автоматика пока ошибается достаточно часто, чтобы без человека не обойтись.

Брутфорс - это не просто много попыток, это конкретный математический паттерн в логах.

На что стоит обратить внимание
1. Аномальная частота (Velocity)
Это самый очевидный признак. Обычный человек ошибается 2–3 раза за минуту. Бот может делать сотни попыток в секунду.

В логах: Мы  увидим пачку событий Event ID 4625 с разницей в миллисекунды.

2. Единый источник (Source IP)
   data.win.eventdata.ipAddress при классическом брутфорсе этот адрес будет одним и тем же для всех сотен неудачных попыток.
Исключение: Если атака распределенная , IP будут разные, но тогда их объединит общая цель (один и тот же targetUserName).

4. Перебор имен (User Enumeration)
Злоумышленники часто не знают точного логина. Они используют словари.

В логах: Мы увидим попытки входа под именами admin, administrator, root, user, test, manager.

4. Паттерн «Password Spraying» (Распыление)
 Чтобы не сработала блокировка аккаунта (например, после 5 ошибок), хакер берет один популярный пароль  и пробует его ко всем пользователям в сети по одному разу.

Характеристика: Много разных targetUserName, но всего по 1–2 ошибки на каждого. Это сложнее поймать обычным правилом-счетчиком.

5. Специфический «Logon Type»
В логах Windows (Event 4625) есть поле LogonType. Оно критически важно для анализа:

Type 2: Кто-то стоит у компа и тыкает в клавиатуру (локальный вход).

Type 3: Сетевой вход (самый частый при брутфорсе через интернет).

Type 10: Удаленный рабочий стол (RDP). Если видим брутфорс с типом 10 - значит,  RDP торчит наружу и его пора прятать.

Напишем правило , которое будем стучать в панельку не столь критическим уровнем, при 5+ неудачных попытках входа
